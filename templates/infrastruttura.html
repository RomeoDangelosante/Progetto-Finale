Protocolli utilizzati

TCP/IP · Livello trasporto

Tutta la comunicazione tra il bot e il server del broker avviene su protocollo TCP/IP, che garantisce la consegna ordinata e affidabile dei pacchetti. Ogni richiesta API è una connessione TCP sulla porta 443 (HTTPS).

HTTPS + TLS 1.3 · Sicurezza

Le comunicazioni sono cifrate con TLS 1.3, il protocollo di sicurezza più recente. Questo protegge i dati in transito — ordini di trading, API key — da eventuali intercettazioni sulla rete.

DNS · Risoluzione nomi

Il bot risolve il dominio del broker (es. api.broker.com) tramite DNS prima di ogni connessione. In caso di problemi DNS la connessione fallisce — è il primo punto di controllo.

HTTP/1.1 + REST · Applicazione

L'API del broker segue l'architettura REST: ogni risorsa (ordini, prezzi, account) ha un endpoint URL dedicato. I metodi HTTP usati sono GET per leggere e POST per scrivere.

Sicurezza — API Key

Autenticazione con API Key

Il bot si autentica al broker tramite una API key univoca. La chiave è:

  • Generata dal broker all'attivazione
  • Salvata in un file .env locale (mai nel codice)
  • Inviata in ogni richiesta come header HTTP
  • Revocabile in qualsiasi momento dal pannello broker

Best practice implementate

  • API key mai scritta nel codice sorgente
  • File .env escluso dal repository (gitignore)
  • HTTPS obbligatorio — nessuna richiesta HTTP non cifrata
  • Timeout sulle richieste per evitare blocchi
  • Gestione errori HTTP con retry automatico
